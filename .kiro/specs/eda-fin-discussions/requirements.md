# Requirements Document

## Introduction

This feature is an Exploratory Data Analysis (EDA) project aimed at identifying and recommending a suitable dataset for predicting whether a stock-related discussion on X/Twitter or Reddit will experience a significant engagement and sentiment surge within the next 24 hours. The project evaluates both publicly available datasets (Kaggle, HuggingFace) and API-based collection feasibility (X/Twitter API, Reddit API), then delivers a recommendation. Primary deliverables are Python scripts and a markdown summary report. An exploration notebook is provided for interactive analysis.

## Glossary

- **EDA_System**: The collection of Python scripts and markdown report that constitute the exploratory data analysis deliverable
- **Public_Dataset_Scanner**: The component responsible for discovering and evaluating publicly available datasets from Kaggle and HuggingFace
- **API_Feasibility_Assessor**: The component responsible for evaluating the feasibility of collecting data via X/Twitter API and Reddit API
- **Surge**: A combined engagement and sentiment surge where a post triggers both high engagement metrics (likes, retweets, comments, upvotes) AND a notable sentiment shift (bullish/bearish change) within 24 hours
- **Engagement_Metrics**: Quantifiable interaction counts including likes, retweets, comments, upvotes, and shares
- **Sentiment_Shift**: A measurable change in sentiment polarity (bullish/bearish) associated with a stock-related discussion
- **Dataset_Report**: The markdown summary document containing EDA findings and a dataset recommendation
- **Chart_Output**: Visualization files (PNG/SVG) generated by the Python scripts during analysis

## Requirements

### Requirement 1: Public Dataset Discovery

**User Story:** As a data scientist, I want to discover and catalog publicly available stock discussion datasets from Kaggle and HuggingFace, so that I can evaluate their suitability for surge prediction modeling.

#### Acceptance Criteria

1. WHEN executed, THE Public_Dataset_Scanner SHALL search Kaggle for datasets containing stock-related social media discussions with engagement metrics.
2. WHEN executed, THE Public_Dataset_Scanner SHALL search HuggingFace for datasets containing stock-related social media discussions with engagement metrics.
3. THE Public_Dataset_Scanner SHALL record for each discovered dataset: name, source platform, number of records, date range, available columns, and data freshness.
4. WHEN a dataset lacks either engagement metrics or sentiment-related fields, THE Public_Dataset_Scanner SHALL flag the dataset as incomplete for surge prediction.
5. IF no datasets are found on a given platform, THEN THE Public_Dataset_Scanner SHALL report the absence and document the search criteria used.

### Requirement 2: API-Based Collection Feasibility Assessment

**User Story:** As a data scientist, I want to understand the feasibility of collecting stock discussion data via X/Twitter and Reddit APIs, so that I can compare API collection against using existing public datasets.

#### Acceptance Criteria

1. THE API_Feasibility_Assessor SHALL evaluate the X/Twitter API for: rate limits, available endpoints, cost tiers, data fields accessible (engagement counts, timestamps, text content), and historical data access.
2. THE API_Feasibility_Assessor SHALL evaluate the Reddit API for: rate limits, available endpoints, cost tiers, data fields accessible (upvotes, comments, timestamps, text content), and historical data access.
3. THE API_Feasibility_Assessor SHALL document the estimated time and cost to collect a minimum viable dataset of 10,000 stock-related posts with engagement metrics from each API.
4. IF an API requires paid access for necessary data fields, THEN THE API_Feasibility_Assessor SHALL document the pricing tier and the specific fields gated behind payment.
5. THE API_Feasibility_Assessor SHALL assess whether each API provides sufficient data to construct the Surge label (combined engagement spike and sentiment shift within 24 hours).

### Requirement 3: Dataset Quality Analysis

**User Story:** As a data scientist, I want to perform exploratory data analysis on the most promising datasets answering key feasibility questions, so that I can determine whether the dataset supports surge prediction and make an informed go/no-go decision.

#### EDA Questions to Answer

The EDA_System SHALL structure its analysis to answer the following questions:

1. Does the dataset support the prediction target (engagement/sentiment surge within 24 hours)?
2. How should surge be defined given the data distributions?
3. Is there enough positive class data (surge events) for model training?
4. Are stock discussions actually different from general discussions in engagement patterns?
5. Can sentiment be extracted reliably from the text content?
6. What features are realistically available for a prediction model?

#### EDA Objectives

The EDA_System SHALL produce findings for each of the following objectives:

A. Dataset structure (schema, column types, record count)
B. Missing values (per-column completeness rates)
C. Time coverage (date range, temporal gaps, posting frequency)
D. Engagement distributions (likes, retweets, comments, upvotes)
E. Sentiment distributions (polarity scores, bullish/bearish ratios)
F. Candidate surge definitions (threshold analysis at multiple percentiles)
G. Class imbalance estimation (positive vs. negative class ratios under candidate definitions)
H. Risks discovered (data quality issues, biases, temporal leakage, coverage gaps)
I. Final dataset decision (go/no-go recommendation with justification)

#### Acceptance Criteria

1. WHEN a candidate dataset is loaded, THE EDA_System SHALL document the dataset structure including: schema definition, column data types, record count, and unique stock ticker count.
2. WHEN a candidate dataset is loaded, THE EDA_System SHALL compute missing value percentages per column and flag columns with more than 30% missing data as high-risk.
3. WHEN a candidate dataset is loaded, THE EDA_System SHALL analyze time coverage including: full date range, identification of temporal gaps exceeding 7 days, and posting frequency distribution.
4. WHEN a candidate dataset is loaded, THE EDA_System SHALL generate distribution charts for engagement metrics (likes, retweets, comments, upvotes) and compute summary statistics (mean, median, percentiles).
5. WHEN a candidate dataset contains text content, THE EDA_System SHALL perform sentiment distribution analysis, generate a sentiment polarity histogram, and report the bullish-to-bearish ratio.
6. THE EDA_System SHALL evaluate candidate surge definitions at multiple engagement percentile thresholds (90th, 95th, 99th) combined with sentiment shift magnitudes (0.5, 1.0, 1.5 standard deviations) and report the resulting positive class count for each combination.
7. THE EDA_System SHALL estimate class imbalance ratios for each candidate surge definition and flag definitions where the positive class represents fewer than 2% of total records.
8. THE EDA_System SHALL compare engagement patterns between stock-specific discussions and general discussions (where data permits) to validate that stock discussions exhibit distinct surge behavior.
9. WHEN text content is available, THE EDA_System SHALL assess sentiment extraction reliability by computing inter-method agreement between at least two sentiment approaches (e.g., lexicon-based and transformer-based).
10. THE EDA_System SHALL catalog all discovered risks including: data quality issues, potential biases, temporal leakage risks, and coverage gaps.
11. THE EDA_System SHALL produce a final dataset decision (go/no-go) with justification based on whether the dataset adequately answers all six EDA questions.
12. IF a candidate dataset fails to support the prediction target on three or more EDA objectives, THEN THE EDA_System SHALL recommend the dataset as unsuitable and document the specific failing objectives.

### Requirement 4: Surge Definition Operationalization

**User Story:** As a data scientist, I want a clearly operationalized definition of "surge" applied to the data, so that I can validate whether the chosen dataset supports the prediction target.

#### Acceptance Criteria

1. THE EDA_System SHALL define a surge as a post where engagement metrics exceed the 95th percentile of the dataset distribution AND sentiment polarity changes by more than one standard deviation from the post's initial sentiment within 24 hours.
2. WHEN computing surge labels, THE EDA_System SHALL use engagement metrics normalized relative to the stock ticker's historical baseline.
3. THE EDA_System SHALL output the percentage of posts labeled as surges in each candidate dataset.
4. IF a dataset does not contain timestamp resolution sufficient to measure 24-hour windows, THEN THE EDA_System SHALL document this limitation and assess whether approximate windowing is feasible.

### Requirement 5: Visualization and Chart Generation

**User Story:** As a data scientist, I want the EDA to produce informative charts saved as files, so that I can include them in reports and presentations without running notebooks.

#### Acceptance Criteria

1. THE EDA_System SHALL save all generated charts as PNG files in a dedicated output directory.
2. WHEN generating charts, THE EDA_System SHALL include descriptive titles, axis labels, and legends.
3. THE EDA_System SHALL generate at minimum: engagement metric distributions, sentiment polarity distributions, surge event frequency over time, and a comparison chart of dataset characteristics across candidates.
4. WHEN a chart is generated, THE EDA_System SHALL log the output file path to standard output.

### Requirement 6: Markdown Report Generation

**User Story:** As a data scientist, I want a comprehensive markdown report summarizing all EDA findings and a final dataset recommendation, so that I have a single document capturing the analysis outcome.

#### Acceptance Criteria

1. THE EDA_System SHALL generate a markdown report file containing: an executive summary, dataset discovery results, API feasibility findings, EDA statistics, surge analysis results, and a final recommendation.
2. THE Dataset_Report SHALL include inline references to generated chart files using relative paths.
3. THE Dataset_Report SHALL provide a clear recommendation of the best data path (public dataset vs. API collection) with justification based on data quality, completeness, cost, and feasibility.
4. WHEN multiple viable options exist, THE Dataset_Report SHALL rank them and explain trade-offs.
5. IF no suitable dataset or API path meets the minimum requirements for surge prediction, THEN THE Dataset_Report SHALL document the gaps and suggest next steps for data acquisition.

### Requirement 7: Project Structure and Execution

**User Story:** As a developer, I want the project organized as executable Python scripts with clear entry points and an exploration notebook for interactive analysis, so that I can run the full EDA pipeline programmatically and also explore data interactively.

#### Acceptance Criteria

1. THE EDA_System SHALL organize core logic as Python scripts (`.py` files) in `src/` with a clear entry point script (`main.py`) that orchestrates the full pipeline.
2. THE EDA_System SHALL include a `requirements.txt` file listing all Python dependencies with pinned versions.
3. WHEN the entry point script is executed, THE EDA_System SHALL run all analysis steps in sequence and produce all outputs (charts and markdown report).
4. THE EDA_System SHALL print progress messages to standard output indicating which analysis step is currently executing.
5. IF a dependency is unavailable or a dataset download fails, THEN THE EDA_System SHALL log a descriptive error message and continue with remaining analysis steps where possible.
6. THE EDA_System SHALL include an exploration Jupyter notebook (`notebooks/exploration.ipynb`) that imports analysis functions from `src/` modules for interactive data exploration and visualization.
7. THE EDA_System SHALL include `jupyter` and `ipykernel` in the `requirements.txt` file.
